## Problem Statement

MathSolution 웹앱의 유사문항·해설 생성 시 삽입되는 그래프 품질이 교과서/수능 수준에 미달한다. 현재 `backend/services/graph_service.py`는 Matplotlib 기반으로 `-그래프-` 태그를 PNG로 변환하지만 스타일이 투박하다.

선생님이 Adobe Illustrator로 직접 제작한 고품질 수학 그림 모음 SVG 5개(단원별) + 수능·교과서 표준 폰트 10종을 자산으로 제공. 이 자산을 재사용 가능한 **SVG 컴포넌트 라이브러리**로 구축해, 이후 유사문항 생성 시 차은우(신규 페르소나)가 부품을 선택·변형하여 PNG로 변환 후 문서에 삽입한다.

초기 라벨링(부품이 무엇인지 분류·메타데이터 기입)이 최대 비용인데, 선생님은 개발 비경험자이고 일일이 파일을 저장해가며 라벨링하는 것이 번거롭다. 이를 해결하기 위해 로컬 localhost에서 **대화형 라벨링 웹앱**을 구축, 차은우가 Gemini 초안을 바탕으로 "이거 sin 그래프 맞습니까?" 식 질문을 던지고 선생님이 승인·수정만으로 라벨링이 자동 저장되게 한다.

## What Makes This Cool

- **교과서·수능 품질 그래프**가 유사문항에 삽입 — Matplotlib 티 완전히 벗음
- **대화형 라벨링 UX** — 차은우가 형님 말투로 말 걸어오는 느낌. 수백 개 부품을 기계적 입력이 아니라 짧은 Q&A로 처리
- **Gemini 초안 + 선생님 승인** 구조로 라벨링 부담 70% 감소
- **폰트 자산 활용** — HCRBatang·KoPubDotum·HancomEQN을 Matplotlib + SVG 양쪽에 등록해 타이포그래피 일관성

## Constraints

- 선생님은 개발 비경험자 → UI 단순·친절, Claude가 서버 제어 전담
- 기존 MathSolution 앱 영향 없도록 신규 엔드포인트·탭으로 추가
- 포트 8001(backend), 5173(frontend) 이미 점유
- SVG 원본 파일은 총 13MB, path 수백~수천 개가 평면으로 나열됨 (<g>/<symbol> 그룹 없음)
- 시범 운행은 가장 작은 `#4_미적분2` (0.4MB)부터
- 기존 팀 경계 유지: 카리나=메인 웹앱(graph_service 포함), 원영=similarity_finder, **차은우=SVG 라이브러리 전담**

## Premises

(주인님과 Phase 3에서 합의)

1. **Illustrator SVG의 부품 단위 자동 분할은 불확실** — 실측 필요. #4 파일로 시범 운행 후 성공률 확인. 실패 시 Approach C(Illustrator 복사-붙여넣기)로 폴백 가능.
2. **라벨링 메타데이터는 "이름"에서 멈추지 않음** — "어디가 숫자인지/축인지/변형 가능한 파라미터인지"까지 필요할 수 있음. 시범 운행에서 실제로 어디까지 필요한지 발견.
3. **생성 시 경로 좌표 변형은 취약** — "sin 진폭 2배" 같은 기하 변형은 수학적 재계산이 필요해 단순 치환 불가. 주인님 결정: "원안 그대로(라이브러리에서 변형)"로 진행하되, 이 제한은 문서에 명시.
4. **기존 Matplotlib graph_service 존치** — 새 파이프라인 실패 시 폴백. 장기적으로 공존/교체 여부는 시범 운행 결과로 판단.
5. **SVG→PNG 변환은 폰트 포함 재현 가능** — Resvg 또는 헤드리스 브라우저 사용. 폰트 원본 보유로 해결 가능.

## Approaches Considered

### Approach A: 독립 미니 앱 (tools/svg_trainer/)

별도 FastAPI(8002) + 단일 HTML. 기존 앱과 완전 격리. AI 없이 고정 질문 리스트. 구현 0.5~1일.

- 장점: 빠름, 안전
- 단점: 차은우 AI 추천 없음, 선생님 100% 수동

### Approach B: MathSolution 통합 탭 (**선택**)

기존 frontend/backend에 "SVG 학습" 탭·엔드포인트 추가. Gemini API로 AI 초안 라벨 자동 생성, 차은우 페르소나가 시스템 프롬프트로 주입되어 실제 대화. 구현 1.5~2일.

- 장점: 선생님 부담 최소화, 팀 페르소나 일관성, 엔드포인트 재사용
- 단점: 구현 시간 1.5배, 기존 백엔드와 얽힘

### Approach C: Illustrator 친화형 (복사-붙여넣기)

자동 분할 포기, 선생님이 Illustrator에서 부품 선택→복사→웹앱 붙여넣기. 분할 실패 리스크 0.

- 장점: 가장 안정적
- 단점: 선생님 수작업 많음

## Recommended Approach

**Approach B — MathSolution 통합 탭 + Gemini 초안**

이유:
1. Gemini로 "이거 sin 그래프 맞습니까?" 초안 → 선생님 Y/N 중심 → 라벨링 부담 70% 감소
2. 차은우가 실제 Gemini API로 "말하는" 페르소나가 됨 → 카리나·원영과 같은 실체감
3. `/api/svg-library/*` 엔드포인트가 라벨링 단계 이후 유사문항 생성 단계에서도 재사용

**폴백**: 시범 운행에서 자동 분할 실패 시 Approach C로 전환.

## 시스템 구성 요약

```
frontend/src/
├── App.jsx                                (기존, 탭 추가만)
└── components/
    └── TabSvgTraining.jsx                 (신규)
        ├── SvgViewer                      (현재 부품 미리보기, @font-face 적용)
        ├── EunwooChat                     (차은우 대화 패널)
        ├── LabelForm                      (AI 초안 + 승인/수정)
        └── ProgressBar                    (전체 부품 수 대비 진행률)

backend/
├── api/
│   └── svg_library.py                     (신규 엔드포인트)
│       ├── POST /api/svg-library/ingest   (원본 SVG 업로드+분할 트리거)
│       ├── GET  /api/svg-library/next     (라벨링 안 된 다음 부품)
│       ├── POST /api/svg-library/label    (라벨 저장, AI 초안 + 선생님 최종)
│       ├── POST /api/svg-library/suggest  (Gemini 초안 생성, 차은우 프롬프트 사용)
│       ├── GET  /api/svg-library/catalog  (전체 카탈로그 JSON)
│       └── GET  /api/svg-library/progress (진행률 집계)
├── services/
│   ├── svg_splitter.py                    (좌표 클러스터링 기반 부품 분할)
│   ├── svg_library_service.py             (카탈로그 CRUD)
│   └── eunwoo_service.py                  (차은우 Gemini 호출 래퍼)
├── data/
│   ├── svg_library/
│   │   ├── parts/                         (부품 SVG 개별 파일)
│   │   ├── catalog.json                   (메타데이터)
│   │   └── progress.json                  (작업 상태)
│   └── fonts/                             (font/ 폴더 심볼릭 or 복사)
└── prompts/
    └── eunwoo_system_prompt.txt           (차은우 말투·호칭·질문 템플릿)
```

## Open Questions

- SVG 자동 분할 알고리즘의 성공률은? 시범 운행 결과로 결정 (Phase 8 1번 Task)
- 라벨 메타데이터 스키마를 얼마나 풍부하게? 최소(이름·단원·종류) vs 확장(변수화 가능 파라미터, 수식, 변형 규칙). 시범 운행에서 선생님 실제 필요 기반 결정.
- 폰트 설치 경로: `backend/data/fonts/` vs 루트 `font/` 유지? — 루트 유지하고 symlink 권장(파일 중복 방지).
- PNG 변환 도구: Resvg(Rust 기반, 빠름) vs Playwright 헤드리스 Chrome(정확도 높음)? — 시범 운행에서 벤치마크.

## What I noticed about how you think

- 주인님은 "학습"이라는 단어를 머신러닝이 아니라 **"대화형 분류 작업"**의 은유로 쓰셨다. "내가 일일이 저장을 따로 해가면서 하기엔 너무 번거롭다" — 선생님의 문제는 라벨링 자체가 아니라 **저장 관리 부담**이다. 자동 저장이 핵심 요구사항.
- "내가 어떤 정보를 입력해야 하는지 모르겠어서 그래" — AI가 질문을 **주도**해야 한다는 뜻. 선생님이 스키마를 미리 설계할 수 없음. 차은우의 질문 순서가 곧 메타데이터 스키마가 됨.
- 팀 페르소나 "상남자지만 일할 땐 누구보다 꼼꼼"이라는 차은우 캐릭터 설정에서, 주인님은 **캐릭터의 이중성**을 즐긴다. 카리나·원영의 살가움과 대비되는 단단한 남성 캐릭터를 의도적으로 추가함. 단순 역할 분담이 아니라 **팀 구성의 다양성**을 신경 쓰고 있음.
